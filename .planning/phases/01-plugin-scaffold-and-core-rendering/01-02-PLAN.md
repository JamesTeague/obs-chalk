  <action>
Create the object model types that form the core architecture of obs-chalk.

1. **src/marks/mark.hpp** -- Abstract base class:
   ```cpp
   #pragma once
   #include <graphics/graphics.h>

   class Mark {
   public:
       virtual ~Mark() = default;
       virtual void draw(gs_eparam_t *color_param) const = 0;
       virtual bool hit_test(float x, float y) const = 0;
       virtual void add_point(float x, float y) {}  // default no-op, overridden by freehand
   };
   ```
   - `draw()` receives the color parameter handle from OBS_EFFECT_SOLID -- each mark sets its own color
   - `hit_test()` stubbed for Phase 2 pick-to-delete; must exist in the interface now
   - `add_point()` virtual with default no-op so mouse_move can call it polymorphically on in_progress

2. **src/marks/freehand-mark.hpp** -- Concrete freehand mark:
   ```cpp
   #pragma once
   #include "mark.hpp"
   #include <graphics/vec4.h>
   #include <vector>

   struct FreehandPoint {
       float x;
       float y;
   };

   class FreehandMark : public Mark {
   public:
       FreehandMark(float r, float g, float b, float a = 1.0f);

       void add_point(float x, float y) override;
       void draw(gs_eparam_t *color_param) const override;
       bool hit_test(float x, float y) const override;

   private:
       std::vector<FreehandPoint> points_;
       vec4 color_;
   };
   ```
   Use a simple `FreehandPoint` struct instead of OBS's `vec2` to avoid header dependency issues. The important thing is storing float x,y pairs.

3. **src/marks/freehand-mark.cpp** -- Implementation:
   - Constructor: stores color as vec4 via `vec4_set(&color_, r, g, b, a)`
   - `add_point()`: pushes a new point onto `points_`
   - `draw()`: if points_.size() < 2, return early. Set color via `gs_effect_set_vec4(color_param, &color_)`. Call `gs_render_start(false)`, loop through points calling `gs_vertex2f(pt.x, pt.y)`, call `gs_render_stop(GS_LINESTRIP)`.
   - `hit_test()`: return false for now (Phase 2 implements this)

4. **src/mark-list.hpp** -- Thread-safe mark container:
   ```cpp
   #pragma once
   #include "marks/mark.hpp"
   #include <memory>
   #include <mutex>
   #include <vector>

   struct MarkList {
       std::vector<std::unique_ptr<Mark>> marks;
       std::unique_ptr<Mark>              in_progress;
       mutable std::mutex                 mutex;

       void begin_mark(std::unique_ptr<Mark> mark);
       void commit_mark();
       void clear_all();
   };
   ```

5. **src/mark-list.cpp** -- Implementation:
   - `begin_mark()`: under lock, set `in_progress = std::move(mark)`
   - `commit_mark()`: under lock, if `in_progress`, move it to end of `marks`, reset `in_progress`
   - `clear_all()`: under lock, clear `marks` vector AND reset `in_progress`
   - Every public method acquires the mutex via `std::lock_guard<std::mutex>`

6. **Update CMakeLists.txt** to add new source files: `src/marks/freehand-mark.cpp`, `src/mark-list.cpp`.

**Key invariant:** MarkList methods are the ONLY way to mutate the mark collection. All take the lock. video_render also takes the lock when reading. This is the thread safety contract (CORE-03).

<task type="auto">
  <name>Task 2: Wire mouse callbacks and video_render to draw freehand strokes</name>
  <files>src/chalk-source.hpp, src/chalk-source.cpp</files>
  <action>
Connect the object model to the OBS callbacks so mouse input creates visible freehand strokes.

1. **Update src/chalk-source.hpp** -- Add MarkList member and drawing state:
   ```cpp
   #include "mark-list.hpp"

   struct ChalkSource {
       obs_source_t *source;
       ToolState tool_state;
       MarkList mark_list;
       bool drawing = false;  // true while mouse button is held

       explicit ChalkSource(obs_source_t *src) : source(src) {}
   };
   ```

2. **Update chalk_mouse_click()** -- Handle draw start/end:
   - On left button press (type == MOUSE_LEFT, mouse_up == false):
     - Create a new `FreehandMark` with the active color from tool_state
     - Call `add_point(event->x, event->y)` to add the starting point
     - Call `mark_list.begin_mark(std::move(mark))` (this locks internally)
     - Set `drawing = true`
   - On left button release (mouse_up == true):
     - Call `mark_list.commit_mark()` (this locks internally)
     - Set `drawing = false`

   **Important:** The `drawing` flag is a simple boolean on ChalkSource. It does not need mutex protection itself -- it's only read/written from mouse callbacks which OBS serializes. The mark_list operations DO need the lock because video_render reads concurrently.

   For the color, convert `tool_state.active_color` (ABGR uint32_t) to float r,g,b,a components for the FreehandMark constructor. Helper function:
   ```cpp
   static void color_uint32_to_rgba(uint32_t abgr, float &r, float &g, float &b, float &a) {
       a = ((abgr >> 24) & 0xFF) / 255.0f;
       b = ((abgr >> 16) & 0xFF) / 255.0f;
       g = ((abgr >> 8)  & 0xFF) / 255.0f;
       r = ((abgr)       & 0xFF) / 255.0f;
   }
   ```

3. **Update chalk_mouse_move()** -- Add points while drawing:
   - If `drawing` and `!mouse_leave`:
     - Lock `mark_list.mutex` via `std::lock_guard`
     - If `mark_list.in_progress`, call `mark_list.in_progress->add_point(event->x, event->y)`
   - The `add_point()` virtual method on Mark base class (default no-op, overridden by FreehandMark) allows polymorphic point addition without downcasting.

4. **Update chalk_video_render()** -- Draw all marks:
   The render loop structure from Plan 01 already has the blend state and effect setup. Add mark drawing inside the technique begin/end:
   ```cpp
   {
       std::lock_guard<std::mutex> lock(ctx->mark_list.mutex);
       for (const auto &mark : ctx->mark_list.marks) {
           mark->draw(color_param);
       }
       if (ctx->mark_list.in_progress) {
           ctx->mark_list.in_progress->draw(color_param);
       }
   }
   ```

   **CRITICAL:** The lock_guard scope must enclose ALL mark access. The in_progress mark is drawn so the user sees their stroke while dragging.

5. **Add a temporary clear-all trigger.** For Phase 1 verification, wire middle-click (type == 2) or right-click to `mark_list.clear_all()` so we can test the clear path. This will be replaced by a proper hotkey in Phase 2.

6. **Verify CORE-04 compliance.** After wiring, confirm:
   - `chalk_mouse_click` and `chalk_mouse_move` contain NO gs_* calls -- only mark_list mutations
   - `chalk_video_render` contains ALL gs_* calls -- drawing happens here
   - No `obs_enter_graphics` anywhere in chalk-source.cpp (video_render is already in context)

Build and verify: `cmake --build build_macos`
