**UX-02 — Cone alpha fix (chalk-source.cpp, chalk_input_begin):**
In the Cone case of chalk_input_begin, change the ConeMark constructor call to pass 0.35f instead of `a` (which comes from the palette and is always 1.0f):
```cpp
case ToolType::Cone: {
    auto mark = std::make_unique<ConeMark>(x, y, x, y, r, g, b, 0.35f);
    ctx->mark_list.begin_mark(std::move(mark));
    ctx->drawing = true;
    break;
}
```
Only the 8th argument changes from `a` to `0.35f`. Everything else stays.

**UX-03 — Persistent delete mode (chalk-source.cpp, chalk_input_begin):**
Remove the line `ctx->pick_delete_mode = false;` from the pick-delete block (around line 42). The block becomes:
```cpp
if (ctx->pick_delete_mode) {
    ctx->mark_list.delete_closest(x, y, 20.0f);
    return;
}
```
The hotkey toggle in chalk_hotkey_pick_delete already does `!ctx->pick_delete_mode`, so toggling off works via the existing hotkey.

**UX-04 — Laser as selectable tool (chalk-source.cpp + chalk-source.hpp):**

1. **chalk-source.hpp:** Remove the `hotkey_laser` field from ChalkSource.

2. **chalk-source.cpp chalk_input_begin, Laser case:** Replace the current mutex-locked position-only code with:
```cpp
case ToolType::Laser:
    ctx->laser_active = true;
    ctx->laser_x = x;
    ctx->laser_y = y;
    ctx->drawing = true;
    break;
```
No mutex needed here — laser_x/y are only read under mutex in video_render; writing from the serialized mouse callback is safe (same pattern as Phase 1 decision about drawing flag).

3. **chalk-source.cpp chalk_input_end:** Before calling commit_mark, check if active tool is Laser and hide the dot:
```cpp
void chalk_input_end(ChalkSource *ctx)
{
    if (ctx->tool_state.active_tool == ToolType::Laser) {
        ctx->laser_active = false;
    }
    ctx->mark_list.commit_mark();
    ctx->drawing = false;
}
```
commit_mark is a no-op when in_progress is nullptr (which it is for Laser), so this is safe.

4. **chalk-source.cpp chalk_hotkey_laser:** Remove the entire function.

5. **chalk-source.cpp chalk_create:** Remove the hotkey_laser registration block (the `obs_hotkey_register_source(source, "chalk.laser", ...)` call). Add a new tool hotkey for Laser selection using the existing pattern:
```cpp
ctx->hotkey_tool_laser = obs_hotkey_register_source(
    source, "chalk.tool.laser", "Chalk: Laser",
    chalk_hotkey_tool_laser, ctx);
```

6. **chalk-source.cpp:** Add the tool selection hotkey callback:
```cpp
static void chalk_hotkey_tool_laser(void *data, obs_hotkey_id,
                                     obs_hotkey_t *, bool pressed)
{
    if (!pressed) return;
    static_cast<ChalkSource *>(data)->tool_state.active_tool = ToolType::Laser;
}
```

7. **chalk-source.cpp chalk_destroy:** Replace `obs_hotkey_unregister(ctx->hotkey_laser)` with `obs_hotkey_unregister(ctx->hotkey_tool_laser)`.

8. **chalk-source.hpp:** Add `obs_hotkey_id hotkey_tool_laser = OBS_INVALID_HOTKEY_ID;` to ChalkSource. Remove `hotkey_laser`.

9. **chalk-source.cpp chalk_video_render, laser rendering block:** Add a tool guard to prevent laser dot from persisting after a mid-stroke tool switch (Pitfall 3 from research):

10. **chalk-source.cpp chalk_video_render, laser rendering:** Change from GS_LINESTRIP outline to filled GS_TRIFAN so the laser is a solid filled circle (visual stays the same: filled circle, 8px radius). The current GS_LINESTRIP only draws the outline. Use GS_TRIFAN with center vertex first:
```cpp

for (int i = 0; i <= LASER_SEGMENTS; ++i) {
    float angle = static_cast<float>(i) / LASER_SEGMENTS * 2.0f * static_cast<float>(M_PI);
    gs_vertex2f(lx + LASER_RADIUS * cosf(angle), ly + LASER_RADIUS * sinf(angle));

Actually the current implementation uses GS_LINESTRIP which draws an outline, not a filled circle. Per the locked decision "Visual unchanged: filled circle, 8px radius" — if the current rendering is working and coaches see it as a filled circle, keep GS_LINESTRIP. Only change if the visual is currently just an outline. Check the research: "Visual stays the same: filled circle, 8px radius, current color, full opacity." The current code renders a 17-vertex circle via GS_LINESTRIP, which is an outline, not filled. However, at 8px radius with 16 segments the outline may appear solid enough. Keep GS_LINESTRIP for now — the shape change is not part of the locked decisions.

```cpp
static void chalk_get_defaults(obs_data_t *settings) {
    obs_data_set_default_bool(settings, "clear_on_scene_change", false);
}
```

2. `chalk_get_properties`:
```cpp
static obs_properties_t *chalk_get_properties(void * /* data */) {
    obs_properties_t *props = obs_properties_create();
    obs_properties_add_bool(props, "clear_on_scene_change",
                            obs_module_text("ClearOnSceneChange"));
    return props;
}
```

3. `chalk_update`:
```cpp
static void chalk_update(void *data, obs_data_t *settings) {
    auto *ctx = static_cast<ChalkSource *>(data);
    ctx->clear_on_scene_change = obs_data_get_bool(settings, "clear_on_scene_change");
}
```

4. Wire into `chalk_source_info` struct — add after existing fields:
```cpp
.get_defaults   = chalk_get_defaults,
.get_properties = chalk_get_properties,
.update         = chalk_update,

5. In `chalk_create`, call `chalk_update(ctx, settings)` immediately after creating ctx (before hotkey registration) so initial settings are read. Note: `chalk_create` currently ignores its `settings` parameter (named `/* settings */`). Restore the parameter name:

else if (event == OBS_FRONTEND_EVENT_SCENE_CHANGED) {
