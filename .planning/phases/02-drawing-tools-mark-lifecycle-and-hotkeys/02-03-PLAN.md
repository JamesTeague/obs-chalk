**Hotkey callback functions** (static, before chalk_create). Each follows the pattern:
```cpp
static void chalk_hotkey_X(void *data, obs_hotkey_id, obs_hotkey_t *, bool pressed) {
    if (!pressed) return;  // fire on key-down only (except laser)
    auto *ctx = static_cast<ChalkSource *>(data);
    // action
}
```

Define these callbacks:
1. `chalk_hotkey_undo` -- calls `ctx->mark_list.undo_mark()` (MARK-01, MARK-04 -- repeated calls step back)
2. `chalk_hotkey_clear` -- calls `ctx->mark_list.clear_all()` (MARK-03)
3. `chalk_hotkey_color` -- increments `ctx->tool_state.color_index = (ctx->tool_state.color_index + 1) % CHALK_PALETTE_SIZE` (INPT-02)
4. `chalk_hotkey_laser` -- special: `ctx->laser_active = pressed` (true on down, false on up -- do NOT early-return on !pressed) (TOOL-05). Laser is independent of active_tool. The laser renders whenever laser_active is true regardless of tool. Just set `ctx->laser_active = pressed`.
5. `chalk_hotkey_tool_freehand` -- sets `ctx->tool_state.active_tool = ToolType::Freehand`
6. `chalk_hotkey_tool_arrow` -- sets `ctx->tool_state.active_tool = ToolType::Arrow`
7. `chalk_hotkey_tool_circle` -- sets `ctx->tool_state.active_tool = ToolType::Circle`
8. `chalk_hotkey_tool_cone` -- sets `ctx->tool_state.active_tool = ToolType::Cone`
9. `chalk_hotkey_pick_delete` -- toggles `ctx->pick_delete_mode = !ctx->pick_delete_mode` (MARK-02 entry)

**chalk_create** -- register all 9 hotkeys using `obs_hotkey_register_source`. Store returned IDs in `ctx->hotkey_*` fields. Use names like `"chalk.undo"`, `"chalk.clear"`, `"chalk.color_next"`, `"chalk.laser"`, `"chalk.tool.freehand"`, `"chalk.tool.arrow"`, `"chalk.tool.circle"`, `"chalk.tool.cone"`, `"chalk.pick_delete"`. Display names: `"Chalk: Undo"`, `"Chalk: Clear All"`, `"Chalk: Next Color"`, `"Chalk: Laser Pointer"`, `"Chalk: Freehand"`, `"Chalk: Arrow"`, `"Chalk: Circle"`, `"Chalk: Cone"`, `"Chalk: Pick to Delete"`.

**chalk_destroy** -- unregister all 9 hotkeys via `obs_hotkey_unregister(ctx->hotkey_*)` BEFORE `delete ctx`.

**chalk_mouse_click** -- rewrite to dispatch on tool state:

Left-click (type 0):
- If `pick_delete_mode` is true and mouse_up is false:
  - Call `ctx->mark_list.delete_closest(event->x, event->y, 20.0f)` (20px threshold)
  - Auto-exit: set `ctx->pick_delete_mode = false`
  - Return early (don't start a new mark)

- If mouse-down (!mouse_up):
  - Get color: `float r, g, b, a; color_uint32_to_rgba(ctx->tool_state.active_color(), r, g, b, a);`
  - Switch on `ctx->tool_state.active_tool`:
    - Freehand: create `FreehandMark(r, g, b, a)`, call `add_point(x, y)`, begin_mark
    - Arrow: create `ArrowMark(x, y, x, y, r, g, b, a)` (tail=head initially), begin_mark
    - Circle: create `CircleMark(x, y, 0.f, r, g, b, a)` (zero radius initially), begin_mark
    - Cone: create `ConeMark(x, y, x, y, r, g, b, a)` (apex=tip initially), begin_mark
    - Laser: no mark created (laser pointer is not a mark)
  - Set `ctx->drawing = true`

- If mouse-up (mouse_up):
  - Call `ctx->mark_list.commit_mark()`
  - Set `ctx->drawing = false`

Right-click (type 2, mouse-down): keep existing `clear_all()` behavior as a convenience.

**chalk_mouse_move** -- rewrite to handle both freehand and drag tools:

If `!mouse_leave`:
  - Update laser position under mutex: `std::lock_guard lock(ctx->mark_list.mutex); ctx->laser_x = x; ctx->laser_y = y;`
    (Laser position is read by video_render on a different thread, so it needs the mutex per research pitfall 3.)

If `ctx->drawing && !mouse_leave`:
  - Lock mutex
  - If in_progress exists:
    - If active tool is Freehand: call `in_progress->add_point(x, y)` (existing pattern)
    - Otherwise: call `in_progress->update_end(x, y)` (drag preview for arrow/circle/cone)

**chalk_video_render** -- add laser pointer rendering inside the technique pass, after drawing all marks:

After the existing mark drawing block, still inside the technique begin/end:
```cpp
if (ctx->laser_active) {
    float lx, ly;
    {
        std::lock_guard<std::mutex> lock(ctx->mark_list.mutex);
        lx = ctx->laser_x;
        ly = ctx->laser_y;
    }
    float r, g, b, a;
    color_uint32_to_rgba(ctx->tool_state.active_color(), r, g, b, a);
    vec4 laser_color;
    vec4_set(&laser_color, r, g, b, 1.0f);
    gs_effect_set_vec4(color_param, &laser_color);
    const float LASER_RADIUS = 8.0f;
    const int LASER_SEGMENTS = 16;
    gs_render_start(false);
    for (int i = 0; i <= LASER_SEGMENTS; ++i) {
        float angle = static_cast<float>(i) / LASER_SEGMENTS * 2.0f * M_PI;
        gs_vertex2f(lx + LASER_RADIUS * cosf(angle),
                    ly + LASER_RADIUS * sinf(angle));
    }
    gs_render_stop(GS_LINESTRIP);
}
```

**chalk-source.hpp** -- No changes needed beyond what Plan 01 already added. If any minor adjustments are required for compilation, make them.

**Critical constraints:**
- All gs_* calls stay inside chalk_video_render (or mark draw() methods called from it). Never in hotkey or mouse callbacks.
- All mark_list mutations from hotkey callbacks (undo_mark, clear_all via hotkey) must NOT hold the mutex explicitly -- those methods already lock internally.
- delete_closest is called from mouse_click (which OBS serializes with mouse_move), but the method locks internally, which is correct since video_render reads from a different thread.
- Laser position update in mouse_move MUST use the mutex since video_render reads laser_x/laser_y.
